In the public directory, we have 4 subdirectories (assets, components, pages and util).

The assets directory contains all the static files, such as css, fonts, images and js files.

The components directory contains all the reusable components, such as the header and footer.

The pages directory contains all the pages of the website, such as the homepage, express, javascriptFundamentals, misc, nodeFundamentals etc...

The util directory contains all the utility files, such as the codeExecutor and typeEffect. These files are used to execute code and create a typewriter effect. Having the utility files in a sperate directory makes it easy to use accross multiple pages and scripts.

Redirection approaches

Server Side Redirection

Server Side Redirections is when the server responds with a HTTP code and a header for 'Location' with the url to redirect to.

Client Side Redirection

In one of the ways with Client Side Redirection is that JavaScript handles it, with the window.location.href assignment, dynamically redirecting the user. 

            
setTimeout(() => (window.location.href = "/publicsquare"), 3000);

In the example above, we set a timeout for 3 seconds from when the page with the script is loaded, to then redirect to '/publicsquare'

Client Side Renderig (CSR) vs Server Side Rendering (SSR)

Client Side Rendering is when a client request a site and the code is sent to the client and then processed and turned into the page we view.

Server Side Rendering is when a client request a site and the server processes the code and sends the page to the client. The client then gets the page instantly.

The difference between the two is that with CSR, the client has to wait for the code to be processed, but with SSR, the client gets the page instantly. However with CSR, the client only has to request the data, not the whole page, which makes the page faster at navigating.

There are several arguments for and against both:

Load Time

SSR
Loads the page faster, as the server sends the full page. However, it can be slower if the server is slow.

CSR
Page loads slower, as the client has to process the code. However, afterwards the page is faster at navigating, because the client only has to request the data, not the whole page.

Resources Spent

SSR
The server handles the processing, so the client doesn't have to, especially if the clients device is slow. However, the server has to be powerful to handle the processing.

CSR
Releases the server of the load of processing the code, but is not ideal for slower networks or devices.

SEO

SSR
Far better for SEO as webcrawlers can read the whole content of the page. It requires resources to render the page for web crawlers and bots.

CSR
Better dynamic content, but not as good for SEO, because it is hard for webcrawlers to read the content.

CORS

SSR
CORS is administered by the browser, so they do not involve browsers with server to server communication. This means that CORS is not a problem with SSR.

CSR
Has to deal with CORS, like setting headers or configuring the backend for allowing CORS.

Code Placement and Application Performance

When declaring and initilazing variables in the global scope, it is available throughout all of your project, which will cause higher memory usage.
Using the local scope will make the code a lot more readable and organizable, and the memory usage will be lower.

Another topic is lazy loading, that is when you only load the modules and scripts when you need them. This will make the initial page load faster. This is especially useful when you have a lot of modules and scripts, and you don't need all of them at the same time.

A thing to always consider is the size of your modules/scripts. If a script is too big, consider splitting it to another file.

Asynchronous programming can also increase perfomance of your application, as you don't block the main thread.

Importance of Clean Code

Clean code is essential, especially when working with others, there has to be some kind of standard in how you write your code.

Code should be written with readability in mind, so that others (and most importantly yourself!) can read and understand it.

It should also be kept simple, so the application doesn't get unnessesarily complex. So chose the most simple way of achieving the desired goal, and avoid overengineering.

It should also be well organized, so that it is easy to navigate and understand.
